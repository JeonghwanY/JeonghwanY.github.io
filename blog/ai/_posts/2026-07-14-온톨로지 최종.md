### Layer 1 - 온톨로지란 무엇인가
정의 = 어떤 도메인에 존재하는 개념(class), 개념 간 관계(property), 그리고 그것들이 지켜야 할 논리적 규칙(axiom)의 명시적 명세.
Gruber의 고전적 정의: "an explicit specification of a conceptualization"(개념화에 대한 명시적인 명세). 여기서 중요한 건 explicit(명시적) 과 shared(공유된) 입니다.
```
Conceptualization (개념화): 우리가 머릿속으로 세상을 바라보는 방식이나 틀.  
예: '인간'이라는 개념 아래에 '이름', '나이', '직업'이 있고, 
인간은 '회사'에 '고용된다'는 관계를 머릿속으로 생각하는 것 자체를 말함.

Explicit Specification (명시적 명세): 머릿속에만 있는 모호한 생각을 컴퓨터가 알아먹을 수 있게 코드로 명확하고 꼼꼼하게 적어두었다는 뜻.
즉, "사람들이 머릿속으로 이해하고 있는 개념과 그 관계들을, 컴퓨터도 
이해할 수 있도록 명확하게 문서(코드)로 정의해 둔 것"이 온톨로지입니다.
```

#### 왜 필요한지 예시

보험 도메인 예시 - 21개 계약정보 파일
파일들에는 아래와 같은 내용이 흩어져 있을 것:
```
파일 A: "계약자는 보험료 납입 의무를 진다"
파일 B: "피보험자 사망 시 수익자에게 보험금 지급"
파일 C: "계약자와 피보험자가 다른 경우 피보험자의 서면동의 필요"
```
Vector RAG는 이걸 텍스트 덩어리의 유사도로만 다룬다. 
"계약자가 보험금을 받나요?"라고 물으면, '계약자'와 '보험금'이 같이 나오는 청크를 긁어온다. 
근데 계약자 ≠ 수익자라는 사실은 청크 안 어디에도 명시적으로 없다.
인간은 문서 3개를 읽고 머릿속에서 구조를 만들지만, 임베딩은 그 역할을 하지 못한다.
온톨로지는 그 구조를 문서 밖으로 꺼내서 명시적으로 쓸 수 있게 만든 것 이다:
```
Class: 계약자, 피보험자, 수익자, 보험계약, 보험금, 보험료
Property: 계약자 --(체결한)--> 보험계약
          보험계약 --(대상은)--> 피보험자
          보험계약 --(지급대상은)--> 수익자
Axiom: 계약자 ≠ 수익자 (반드시 같지는 않음)
       피보험자 ≠ 계약자 인 경우 → 서면동의 필요
```
이게 바로 온톨로지. 파일이 몇 개든, 문장이 어떻게 쓰였든, 도메인의 뼈대(설명하는 방식[문장, 언어, 파일 형식]은 수백 가지로 달라져도, 그 뒤에 숨어 있는 진짜 규칙과 정답은 단 하나)는 하나라는 게 핵심 아이디어 이다.

#### 온톨로지의 3요소
| 요소 | 뜻 | 보험 예 |
|------|------|----------|
| **Class (개념)** | 사물의 종류 | 보험계약, 계약자, 특약 |
| **Property (관계/속성)** | 사물 간 연결 또는 사물의 값 | 체결한 (관계), 납입보험료 (값) |
| **Axiom (공리)** | 논리적 규칙 | 종신보험 ⊑ 생명보험 (모든 종신보험은 생명보험이다) |

Class vs Instance 구분이 결정적입니다:

보험계약 = Class (개념, 틀)
계약번호 2024-XXXX = Instance (실제 개별 사례)

온톨로지는 주로 Class 층을 다루고, 실제 데이터(Instance)는 그 틀에 채워 넣습니다. Class 층을 TBox (Terminological(터미널러지컬) Box), Instance 층을 ABox(어썰쎠널) (Assertional Box)라고 부릅니다.

### Layer 2 — RDF

RDF는 세상의 모든 지식을 **3개짜리 문장**으로만 표현한다.

```turtle
(Subject, Predicate, Object)
 주어      서술어      목적어
:계약_2024_001  :계약자   :홍길동 .
:계약_2024_001  :피보험자  :김철수 .
:계약_2024_001  :보험료   "50000"^^xsd:integer .
```

이 3개 트리플이 모이면 자동으로 **그래프**가 된다. 
Subject와 Object가 노드, Predicate가 엣지이기 때문입니다.

```
[계약_2024_001] --계약자--> [홍길동]
       |
       +--피보험자--> [김철수]
       |
       +--보험료--> ("50000")
```

**RDF에서 "그래프"는 설계한 게 아니라 트리플을 던져 넣으니 저절로 생긴 것.** 스키마를 먼저 정의하고 테이블을 만드는 게 아닙니다. 이게 LPG와 다른점.

#### URI — 왜 `:홍길동`이 아니라 이상한 주소인가

실제 RDF에서 모든 것은 **URI**로 식별된다.

```turtle
@prefix aia: <http://aia.co.kr/ontology#> .

aia:계약_2024_001  aia:계약자  aia:홍길동 .
```

여기서 `aia:계약자`는 실제로 `http://aia.co.kr/ontology#계약자`의 축약형입니다.

전 세계 어디서든 이 개념을 **하나로 특정하기 위해서**이런 구조를 가지게 됨. 
회사 A의 "계약자"와 회사 B의 "계약자"가 같은 개념인지 다른 개념인지, 문자열 "계약자"만으로는 알 수 없다. 
URI가 다르면 다른 개념, 같으면 같은 개념입니다. 이것이 RDF가 태어난 목적(Semantic Web = 웹 전체를 하나의 DB로)의 직접적 결과이다.

실제 웹주소는 아님. 전역 고유 이름표일 뿐. 

#### Literal vs Resource

Object 자리에는 두 가지가 올 수 있다:

```turtle
aia:계약_001  aia:계약자   aia:홍길동 .          
# Resource — 다른 노드로 연결됨
aia:계약_001  aia:보험료   "50000"^^xsd:int .    
# Literal — 값, 여기서 끝. 더 못 나감
```

**Literal(리터럴)은 그래프의 잎(leaf)이다.** Literal을 Subject 자리에 놓을 수 없다. 
`"50000" :단위 "원" .` 은 RDF에서 **불법**이다. 
[이건 나중에 LPG와 비교할 때 결정적으로 중요해진다.]
이게 큰 단점이 되는 경우는 실무에서 개발할 때, "어떤 데이터에 추가적인 세부 정보를 또 붙이고 싶을 때" 굉장히 번거롭다.
RDF에서는 이 규칙을 우회하기 위해 세부 정보를 붙이고 싶은 개념을 리터럴(값)이 아닌, 고유한 주소(IRI)를 가진 주체(노드)로 격상시키켜야 한다.

요약하자면
진짜 '값(리터럴)'은 오직 오브젝트(Object)만 가능하다.

만약 그 값에 추가적인 속성이나 관계를 붙인다면, RDF에서는 그 값을 주소(IRI)를 가진 독립된 노드로 설계해야 한다.

이 제약 때문에 RDF는 모델링이 엄격해지고 데이터양이 늘어나는 경향이 있지만, 모든 것이 주어-서술어-목적어로 완벽하게 통일되기 때문에 컴퓨터가 복잡한 논리를 자동으로 추론(Reasoning)하기에는 훨씬 유리하다.

#### Turtle 문법

실제로 읽고 쓸 수 있어야 하는 문법:

```turtle
@prefix aia: <http://aia.co.kr/ontology#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .

# 마침표 = 트리플 끝
aia:계약_001 rdf:type aia:보험계약 .

# 세미콜론 = 같은 Subject 반복
aia:계약_001 
    rdf:type      aia:보험계약 ;
    aia:계약자     aia:홍길동 ;
    aia:피보험자   aia:김철수 ;
    aia:보험료     "50000"^^xsd:integer .

# 쉼표 = 같은 Subject + 같은 Predicate 반복
aia:계약_001 aia:특약 aia:암특약, aia:재해특약 .

# 'a' 는 rdf:type 의 축약형 (자주 나옴)
aia:계약_001 a aia:보험계약 .
```

`;` `,` `.` 세 개를 구분한다면  Turtle을 읽을 수 있다.

### Layer 3 — RDFS / OWL (온톨로지를 RDF로 쓰기)

RDF는 "트리플을 쓸 수 있다"까지만 안다. 
**어떤 트리플을 써야 온톨로지가 되는가**를 정한 것이 RDFS와 OWL이다.

#### RDFS — 최소한의 어휘

```turtle
# 클래스 선언
aia:보험계약  a  rdfs:Class .
aia:종신보험  a  rdfs:Class .

# 클래스 계층 (상속)
aia:종신보험  rdfs:subClassOf  aia:생명보험 .
aia:생명보험  rdfs:subClassOf  aia:보험계약 .

# 프로퍼티 선언 + domain/range
aia:계약자  a  rdf:Property ;
    rdfs:domain  aia:보험계약 ;    # 이 관계의 출발점은 보험계약
    rdfs:range   aia:사람 .        # 이 관계의 도착점은 사람
```

#### 중요한 함정 — domain/range는 제약이 아니다

**RDBMS 감각으로 보면 `rdfs:range aia:사람`은 "사람이 아니면 에러"로 읽을 수 있지만 아니다.**

RDF/OWL은 **Open World Assumption (OWA)** 위에서 돌아간다. 그래서 이렇게 동작한다:

```turtle
# 온톨로지에 이렇게 선언되어 있고
aia:계약자  rdfs:range  aia:사람 .

# 데이터로 이걸 넣으면
aia:계약_001  aia:계약자  aia:법인_ABC .

# 결과: 에러 아님. 대신 추론기가 이렇게 결론냄:
aia:법인_ABC  a  aia:사람 .    # ← 자동으로 추론됨!
```

**"법인ABC도 사람인가 보네"라고 유연하게 결론을 내린다.** 
이게 OWA이지만 오타나 너무 터무니없는 값이 들어온다면 오류가 발생하기 때문에 검증이 필요하다.

| | Closed World (SQL) | Open World (RDF/OWL) |
|---|---|---|
| 없는 데이터 | 거짓(false) | 모름(unknown) |
| 스키마 위반 | 에러 | 추론 재료 |
| `rdfs:range` | 제약 | **추론 규칙** |

**그럼 진짜 검증(validation)은 어떻게 하나?** → **SHACL**. RDF 데이터에 "이건 반드시 지켜야 함"을 걸고 위반을 리포트하는 별도 표준이다. 
실무에서 데이터 품질을 잡으려면 이 SHACL이 필요하다. RDFS/OWL로는 안 됩니다. 

#### OWL — 논리로 더 세게

OWL은 RDFS 위에 표현력을 얹는다:

```turtle
# 두 개념은 서로 배타적
aia:개인계약  owl:disjointWith  aia:단체계약 .

# 역관계 — 하나 넣으면 반대 방향 자동 생성
aia:피보험자  owl:inverseOf  aia:보험대상계약 .

# 이행성 — A>B, B>C 면 A>C
aia:포함하는특약  a  owl:TransitiveProperty .

# 동일 개체 선언 (다른 파일에서 온 같은 사람)
aia:홍길동  owl:sameAs  legacy:CUST_00123 .
```

**추론(reasoning)의 실체:** 추론기(Pellet, HermiT, RDFox 등)를 돌리면, 명시적으로 안 쓴 트리플이 **새로 생성**됩니다.

```
입력: 계약_001 a 종신보험 .
      종신보험 rdfs:subClassOf 생명보험 .
      생명보험 rdfs:subClassOf 보험계약 .

추론 결과 (자동 생성):
      계약_001 a 생명보험 .
      계약_001 a 보험계약 .
```

**이것이 RDF가 LPG 대비 갖는 유일하고 진짜인 강점입니다.** "종신보험 계약 다 찾아줘"가 아니라 "생명보험 계약 다 찾아줘"라고 물어도 종신보험이 나옵니다. 계층을 쿼리에 하드코딩할 필요가 없습니다.
```
예시
[RDF 규칙 (온톨로지 뼈대)]
* 1단계: "종신보험"은 "생명보험"의 일종이다. (subClassOf)
* 2단계: "생명보험"은 "보험계약"의 일종이다. (subClassOf)

[우리가 입력한 실제 데이터]
* "계약_001"은 "종신보험"이다. 

-> 를 바탕으로 추론기는 뇌를 굴려 꼬리에 꼬리를 물고 계층 구조를 타고 올라간다.

"계약_001이 종신보험이네?"
"규칙을 보니 종신보험은 전부 생명보험의 하위 개념이군"
"그럼 계약_001은 생명보험이겠군!" ──> 계약_001 a 생명보험 자동 생성됨
"근데 생명보험은 또 보험계약의 하위 개념이네?"
"그러면 계약_001은 보험계약도 되겠군" ──> 계약_001 a 보험계약 자동 생성됨

1.실무에서 '미친 듯한 강점'인 이유.
일반적인 데이터베이스(RDB나 일반 LPG)를 쓴다면, 
사용자가 "생명보험 계약 다 보여줘"라고 요청했을 때 
개발자는 쿼리를 복잡하게 짜야 한다.

일반 DB (추론 없음)의 경우
컴퓨터는 "종신보험"과 "생명보험"이 무슨 관계인지 모른다. 
그래서 개발자가 쿼리에 하드코딩을 해야한다.
-> "DB야, 생명보험 계약 다 찾아주고 종신보험, 암보험, 정기보험도 
전부 생명보험 종류니까 그것들도 같이 다 찾아서 가져와"

나중에 새로운 보험 상품인 'AI종합건강보험'이 추가된다면 개발자는 소스코드를 열어서 쿼리를 또 수정해야 한다.

RDF/OWL (추론 있음)의 경우
이미 추론기가 계약_001에 "생명보험"이라는 태그를 내부적으로 자동으로 붙여 놓은 것.
-> 개발자가 심플하게 질문만 던져도 잘 가져온다.

"DB야, 생명보험 계약 다 찾아줘."
-> 컴퓨터는 종신보험인 계약_001을 알아서 슥 건네준다. 
이미 추론 결과로 계약_001 a 생명보험이라는 관계가 만들어졌기 때문.

3. 요약
대장 상위 개념들(생명보험, 보험계약)과의 관계를 컴퓨터가 알아서 스스로 계산해서 만들어내는 것이 추론이다.

이 덕분에 데이터가 아무리 복잡하게 얽혀 있어도, 
개발자가 쿼리에 상속 관계를 일일이 하드코딩할 필요가 없어진 것 이다.
신규 데이터가 추가되어도 뼈대 규칙만 타면 
알아서 분류가 가능한 것이 RDF구조의 장점이다.
```

#### SPARQL 최소 세트

```sparql
PREFIX aia: <http://aia.co.kr/ontology#>

SELECT ?계약 ?계약자이름
WHERE {
	#'생명보험' 카테고리에 속하는 '?계약'들을 다 가져와.
    ?계약  a          aia:생명보험 ;
    
    # 방금 찾은 그 '?계약'의 '계약자'인 '?사람'을 찾아.
           aia:계약자  ?사람 .
           
    # 그 '?사람'의 '이름'인 '?계약자이름'을 알아내.
    ?사람  aia:이름    ?계약자이름 .
    
    #'?계약자이름'이 "홍길동"인 녀석은 탈락시켜.
    FILTER(?계약자이름 != "홍길동")
}
LIMIT 10
```

**읽는 법:** `WHERE {}` 안은 **트리플 패턴**이다. `?로 시작하면 변수`. 위 쿼리는 "이 모양의 트리플들이 그래프 안에 존재하면, 거기 맞는 값을 뽑아라"이다. **SQL의 JOIN이 아니라 패턴 매칭.**

### Layer 4 — LPG (Labeled Property Graph)

Neo4j가 쓰는 모델이고 RDF와 **근본적으로 다르다.**

#### 구조

```
Node:
  - 여러 개의 Label 가질 수 있음: :보험계약, :생명보험
  - 내부에 Property(key-value) 를 직접 품음: {계약번호: "2024-001", 보험료: 50000}

Relationship:
  - 정확히 하나의 Type: :계약자
  - 방향 있음
  - ★★ Relationship 자신도 Property 가질 수 있음: {체결일: "2024-01-01"}
```

## RDF vs LPG — 결정적 차이

**RDF의 엣지(predicate)는 속성을 가지지 못한다.** 트리플은 3칸뿐이기 때문.

"홍길동이 계약_001을 **2024년 1월 1일에** 체결했다"를 RDF로 쓰려면:

```turtle
# 이건 불가능:
aia:홍길동 aia:체결 aia:계약_001 [체결일 "2024-01-01"] .   # ← RDF에 이런 문법 없음

# 관계를 노드로 승격시켜야 함 (Reification / n-ary relation):
aia:체결사건_001  a          aia:체결사건 ;
                  aia:주체    aia:홍길동 ;
                  aia:대상    aia:계약_001 ;
                  aia:체결일  "2024-01-01" .
```

트리플 1개가 4개로 늘어나는 RDF의 고질적인 불편함.

LPG는 그냥:
```cypher
(홍길동)-[:체결 {체결일: "2024-01-01"}]->(계약_001)
```

#### 비교표 

| | **RDF** | **LPG (Neo4j)** |
|---|---|---|
| 원자 단위 | 트리플 (S,P,O) | Node + Relationship |
| 엣지에 속성 | 불가 (reification 필요) | 네이티브 |
| 노드 속성 | 별도 트리플로 표현 | 노드 내부에 저장 |
| 식별자 | 전역 URI | 내부 ID (로컬) |
| 스키마 | RDFS/OWL (표준, 형식논리) | 없음 (관례 or APOC/plugin) |
| **추론** | 표준 추론기 존재 | **없음** (직접 구현) |
| 검증 | SHACL | 제약조건(unique, exists) 수준 |
| 쿼리 | SPARQL (패턴매칭) | Cypher (패턴매칭) |
| 개방성 | Open World | Closed World |
| 데이터 통합 | 강함 (URI merge) | 약함 |
| 순회 성능 | 상대적으로 느림 | 빠름(인덱스프리 인접성)|
| 표준화 | W3C 표준 | Cypher는 GQL(ISO)로 표준화 진행 |

#### 실무적인 결론

Neo4j에는 OWL 추론기가 없다. `종신보험 subClassOf 생명보험`을 넣어도, Neo4j는 그걸 **아무 의미 없는 그냥 엣지**로 본다. "생명보험 다 찾아줘"라고 하면 종신보험은 나오지 않는다.

Neo4j에서 계층 추론을 하기 위해선 **쿼리에 직접 써야함**:

```cypher
MATCH (c)-[:IS_A*0..]->(:Class {name: "생명보험"})
RETURN c
```

`*0..`-> "0단계 이상 아무 depth나 타고 올라가라"이다. 

**추론을 쿼리로 흉내내는 것**. 
그러나 OWL처럼 disjointWith, inverseOf, TransitiveProperty를 자동 처리해주지는 않는다.

**neosemantics(n10s)** — RDF를 Neo4j로 임포트하고 제한적 추론을 제공하는 공식 플러그인이 존재한다.

### Layer 5 — Neo4j는 물리적으로 어떻게 도는가?
Neo4j의 가장 핵심적인 설계 목표는 "디스크 IO(하드디스크 읽기/쓰기)를 0(zero)에 가깝게 만드는 것"

#### 핵심 개념: Index-Free Adjacency (IFA)

**RDBMS에서 그래프식의 순회가 느린 이유:**

```sql
-- "홍길동의 친구의 친구" 
SELECT ... FROM person p1
JOIN friend f1 ON p1.id = f1.from_id     -- 인덱스 B-tree 탐색: O(log N)
JOIN person p2 ON f1.to_id = p2.id       -- 또 O(log N)
JOIN friend f2 ON p2.id = f2.from_id     -- 또 O(log N)
JOIN person p3 ON f2.to_id = p3.id       -- 또 O(log N)
```

**JOIN 할 때마다 인덱스를 찾는다.** depth가 깊어질수록 비용이 곱해지고 데이터가 커지면(N이 커지면) 매 hop이 다 느려진다.

**Neo4j의 방식:**

노드가 **자기 이웃의 물리적 주소(디스크 offset)를 직접 들고 있다.**

```
Node 레코드 (고정 15바이트):
  [inUse][nextRelId][nextPropId][labels][extra]
                ↑
        이 노드에 붙은 첫 번째 관계의 ID

Relationship 레코드 (고정 34바이트):
  [inUse][firstNode][secondNode][relType]
  [firstPrevRelId][firstNextRelId]      ← firstNode 입장에서의 이전/다음 관계
  [secondPrevRelId][secondNextRelId]    ← secondNode 입장에서의 이전/다음 관계
  [nextPropId][firstInChain]
```

관계 레코드들이 **이중 연결 리스트**를 이룬다. 
노드에서 출발해 관계 체인을 따라가면, **인덱스 탐색 없이 포인터만 따라가면 되서 매우 빠르고 정확하다.**

**레코드가 고정 크기라는 게 핵심.** ID를 알면 디스크 offset을 **곱셈 한 번으로** 계산한다:

```
offset = record_id × 15    (노드)
offset = record_id × 34    (관계)
```

**O(1) 접근으로 탐색이 없다.**

#### 결과: 성능 특성

| | RDBMS JOIN | Neo4j IFA |
|---|---|---|
| 1 hop 비용 | O(log N) — N=전체 데이터 크기 | **O(1)** — 전체 크기와 무관 |
| k hop 비용 | O(k · log N) | O(k · d) — d = 평균 차수 |
| 데이터 증가 시 | 느려짐 | **로컬 순회는 안 느려짐** |

**"Neo4j의 IFA는 로컬 순회에서 전체 데이터 크기에 무관한 성능을 낸다."**

#### LPG의 한계

1. **첫 노드를 찾을 때는 인덱스를 사용.** `MATCH (p:Person {name:"홍길동"})` — B-tree/Lucene 인덱스 탐색이고 IFA는 **그 다음부터**적용됨.
2. **Supernode 문제.** 관계가 수백만 개 붙은 노드(예: `:보험상품` 노드에 모든 계약이 연결)는 연결 리스트가 너무 길어서 순회가 느려진다. 
Neo4j는 이걸 완화하려고 관계를 type/방향별로 그룹핑(relationship group)하지만, 근본 해결으로 보이진 않는다. 모델링으로 해결?..
3. **집계/스캔은 RDBMS가 우세하다.** "전체 계약 평균 보험료"같은 건 그래프 DB의 강점이 아니다.

#### 저장 파일 구조
Neo4j는 데이터를 하나의 파일에 모두 저장하지 않고, 역할(노드, 관계, 속성 등)에 따라 파일을 칼같이 분리하여 저장한다.

```
neostore.nodestore.db          ← 노드 레코드
:오직 노드(Node)의 뼈대 정보만 담고 있는 파일이고 고정 크기(15바이트) 레코드로 구성되어 있어 주소 계산이 매우 빠르다.

neostore.relationshipstore.db  ← 관계 레코드
:누가 누구와 연결되어 있는지, 관계(Relationship)의 연결 정보만 고정 크기(34바이트)로 담는다.

neostore.propertystore.db      ← 프로퍼티
:노드와 관계에 들어있는 키-값 속성(Property) 중, 크기가 작고 정형화된 데이터(숫자, 짧은 텍스트 등)를 저장한다.

neostore.propertystore.db.strings  ← 긴 문자열
:속성의 크기가 너무 커서 속성 상자에 다 안 들어갈 때 그 긴 문자열만 이 파일에 따로 떼어놓고 저장한다.

neostore.labelscanstore.db     ← 라벨 인덱스
:"여기서 :고객 라벨을 가진 노드가 다 어디 있더라?" 하고 빠르게 찾을 때 쓰는 색인(Index) 파일이다.
```

+ 페이지 캐시 (JVM 힙 밖, off-heap)에 이 파일들을 올려놓고 사용한다. **그래프 전체가 RAM에 들어가면 성능이 극대화됩니다.** Neo4j 사이징의 제1원칙(서버메모리를 충분히 구매)입니다.

### Layer 6 — Neptune

AWS 관리형 그래프 DB로 **RDF와 LPG를 둘 다 지원한다.**

| | Neptune |
|---|---|
| 모델 | RDF **또는** Property Graph |
| 쿼리 | SPARQL 1.1 / Gremlin / **openCypher** |
| ★ 제약 | **두 모델의 데이터는 서로 안 보임.** RDF로 넣은 걸 Gremlin으로 못 읽음 |
| 추론 | **OWL 추론기 없음** |
| 특징 | 관리형, 스토리지 자동확장, 읽기 복제본 |
| Neptune Analytics | 별도 서비스. 그래프 알고리즘 + **벡터 검색** 지원 |

**중요한 사실 2가지:**

1. **Neptune은 OWL 추론을 안한다.** SPARQL은 쓸 수 있지만 추론기는 없다. 추론이 필요하면 GraphDB, Stardog, RDFox 같은 걸 사용해야 한다.
2. **Neptune Analytics가 벡터 검색을 지원합니다.** Graph RAG 관점에서 이게 AWS가 미는 지점이다. AIA가 AWS 스택일까?(Bespin이 AWS 파트너임이 확인되긴 함)

### Layer 7 — Graph RAG 연결

#### Graph RAG는 두 갈래로 나뉜다

##### (A) Microsoft GraphRAG 방식 — 문서에서 그래프를 뽑기

```
문서 → LLM으로 엔티티/관계 추출 → 그래프 구축 
     → Leiden 알고리즘으로 커뮤니티 탐지 
     → 각 커뮤니티를 LLM이 요약 
     → 질문 시 커뮤니티 요약들을 계층적으로 활용
```

**푸는 문제:** "이 문서 전체의 주요 테마가 뭐야?" 같은 **global 질문**. Vector RAG는 top-k 청크만 보니까 전체 조망이 불가능하다.

**온톨로지 없어도 동작.** LLM이 알아서 엔티티를 뽑는다. 대신 **일관성이 없어진다** ("계약자" / "보험계약자" / "가입자"가 다 다른 노드가 됨).

##### (B) 온톨로지 기반 KG 방식 — 스키마 먼저

```
도메인 온톨로지 설계 (사람이) 
     → 문서에서 그 온톨로지에 맞춰 인스턴스 추출 (LLM이) 
     → KG 구축 
     → 질문을 SPARQL/Cypher로 변환 → 정확한 사실 조회
```

**푸는 문제:** **정확성이 필수인 질의.** "계약자와 피보험자가 다를 때 필요한 절차는?" — 이건 유사도 검색이 아니라 **구조를 타고 가야** 답변이 나온다.

**보험 도메인은 (B)로 가는게 맞음.** 규제 산업이고, 답이 틀리면 안 되고, 도메인 구조가 이미 법으로 명확히 정의되어 있다(보험업법, 약관 표준).

#### 실무 아키텍처 — (B)에 Hybrid(Neo4j)가 정답

```
질문
 ├─→ Vector Search  → 관련 문서 청크 (근거 원문)
 └─→ Graph Query    → 구조적 사실 (관계, 계층, 제약)
              ↓
        둘 다 컨텍스트로 LLM에 투입
              ↓
           최종 답변
```

**둘 중 하나를 고르는 게 아닌 둘다 사용.** 벡터는 "원문 근거"를, 그래프는 "구조적 사실"을 제공한다. 보험 문서 QA에서는 둘 다 필요하다고 판단했다.

### AIA 계약정보 21개 파일 — 구체적 실행 계획

#### 0. 왜 Graph RAG인가 — Vector RAG가 깨지는 지점

온톨로지를 도입하기 전에, **현재 벡터 RAG가 실제로 틀리는 것**을 먼저 증명해야 한다. 근거 없는 도입은 유행 추종이다.

| 실패 유형 | 예시 질문 | 벡터가 깨지는 이유 |
|---|---|---|
| **Multi-hop** | "종신보험에 암특약 붙인 계약의 부활 조건은?" | 답이 3개 문서에 흩어져 있고, 그것들이 서로를 언급하지 않음 |
| **역질의/집계** | "이 조건에 해당하는 특약을 전부 나열해줘" | top-k는 완전성을 보장하지 않음. 6번째로 유사한 특약은 누락됨 |
| **관계 구분** | "계약자가 사망하면?" vs "피보험자가 사망하면?" | 임베딩 공간에서 거의 같은 벡터. 법적 효과는 정반대 |
| **계층 추론** | "생명보험 전체에 적용되는 규정은?" | 문서에 "종신보험"이라 써 있으면 "생명보험" 질의에 안 걸림 |
| **부정/예외** | "어떤 경우에 보험금이 지급되지 않는가?" | 면책조항은 지급조항과 텍스트가 유사. 부정을 임베딩이 못 잡음 |

**→ 이 5개를 실제로 돌려서 실패 로그를 첨부한다. 이게 도입 근거의 전부다.**

#### Step 1. Competency Question

온톨로지 공학의 표준 방법론. **"이 온톨로지가 답할 수 있어야 하는 질문 목록"** 을 먼저 확정한다.

**CQ가 온톨로지의 스펙이다.**
- CQ에 답 못 하면 그 온톨로지는 실패다.
- **CQ에 없는 개념은 넣지 않는다.** 범위를 CQ가 정한다.

```
CQ1.  특정 계약이 특정 계약변경(감액/납입주기변경/보종전환 등)을 신청 가능한 상태인가?
CQ2.  이 계약변경을 신청하려면 누가(계약자 본인/대리인/법정대리인) 어떤 서류를,
      어떤 채널로 제출해야 하는가?
CQ3.  이 계약변경 건에 타인을위한해지프로세스가 적용되는가? (계약자와 수익자가 상이한 경우)
CQ4.  이 계약변경에서 본인인증이 필요한 조건은 무엇이고, 요구되는 인증방식(휴대폰/신분증)은?
CQ5.  실효된 계약이 특정 시점 기준 자동부활/일반부활 대상인가, 그 조건과 기한은?
CQ6.  특정 계약상태(정상/실효/납입면제 등)에서 이 계약변경이 허용되는가, 불허되는가?
CQ7.  이 계약변경이 AML/FATCA-CRS 대상에 해당하는가, 해당 시 추가 구비서류는?
CQ8.  미성년자/법인/외국인 계약자인 경우 이 업무에서 추가로 필요한 절차·서류는?
CQ9.  이 계약변경 처리로 반제금/추징금이 발생하는가, 발생 시 지급기준(지급일/반환계좌 제약)은?
CQ10. 이 계약변경이 비교안내 대상인가? (신계약 후 6개월 이내 등 조건)
CQ11. 특정 통지(미납안내/해지예고/실효안내)가 어떤 수신자·전달매체를 거쳤을 때
      법적으로 유효한 수령/안내완료로 인정되는가?
CQ12. 특정 업무 처리가 완료된 후 취소가 가능한가, 가능하다면 기한과 조건은?
CQ13. 특정 업무 처리가 다른 특약/보장/부가서비스(멤버십, 건강관리자금, 연금전환 등)에
      어떤 파급 효과를 미치는가?
CQ14. 청약 시점과 현재 시점의 상태(국적, 건강상태, 직업 등)가 다를 경우
      어느 시점 기준으로 자격/가능여부를 판단하는가?
```

#### Step 2. CQ 유형 분류 — **데이터 모델 선택의 근거**

**이 표가 전체 설계 결정의 핵심 근거다.** "Neo4j를 왜 쓰냐"에 대한 답이 여기서 나온다.

CQ를 **어떤 종류의 그래프 연산을 요구하는가**로 분류한다:

| 유형 | 정의 | 필요한 것 |
|---|---|---|
| **① 조건부 규칙** | IF 조건들 THEN 판정. 다수 속성의 조합 평가 | 규칙 평가 (OWL로는 어려움 → SWRL/SHACL 필요) |
| **② 계층 추론** | 상위/하위 개념 자동 확장 | `rdfs:subClassOf` + 추론기 **또는** 가변길이 패턴 |
| **③ 다중홉 순회** | 관계를 여러 단계 타고 이동 | 그래프 순회 (RDF/LPG 공통 강점) |
| **④ 관계 속성** | 엣지 자체에 조건/시점/근거가 붙음 | LPG 네이티브 / RDF는 reification 필요 |
| **⑤ 시점 의존** | 같은 사실이 시점에 따라 다름 | 시간 모델링 (양쪽 다 별도 설계 필요) |

**분류 결과:**

| CQ | 핵심 질문 성격 | ① 조건규칙 | ② 계층추론 | ③ 다중홉 | ④ 관계속성 | ⑤ 시점 |
|---|---|:---:|:---:|:---:|:---:|:---:|
| CQ1 | 계약상태 + 상품유형 → 변경가능 여부 | ● | ○ | | | |
| CQ2 | 변경유형 → (주체, 서류, 채널) 조합 | ● | | ● | ● | |
| CQ3 | 계약자≠수익자 → 프로세스 분기 | ● | | ● | | |
| CQ4 | 변경유형 + 금액/채널 → 인증방식 | ● | | | ● | |
| CQ5 | 실효 + 경과일 + 완납 → 부활유형 | ● | | | | ● |
| CQ6 | 계약상태 × 변경유형 → 허용/불허 매트릭스 | ● | ○ | | ● | |
| CQ7 | 계약속성 → AML/FATCA 대상 판정 → 서류 | ● | | ● | | |
| CQ8 | 계약자 유형(미성년/법인/외국인) → 추가절차 | ● | ● | ● | | |
| CQ9 | 변경 처리 → 금전 발생 → 지급기준 | ● | | ● | ● | |
| CQ10 | 신계약 후 경과기간 → 비교안내 대상 | ● | | | | ● |
| CQ11 | (통지, 수신자, 매체) → 법적 유효성 | ● | | | ● | |
| CQ12 | 처리완료 → 취소가능 여부 + 기한 | ● | | | ● | ● |
| CQ13 | 처리 → 다른 특약/서비스로 **파급 전파** | | ○ | **●** | ● | |
| CQ14 | 시점(청약/현재) → 기준 선택 | ● | | | | **●** |

**● = 핵심 요구, ○ = 부수적 요구**

**집계:**

| 유형 | 해당 CQ 수 | 비율 |
|---|---|---|
| ① 조건부 규칙 | **13 / 14** | **93%** |
| ③ 다중홉 순회 | 6 / 14 | 43% |
| ④ 관계 속성 | 7 / 14 | 50% |
| ⑤ 시점 의존 | 4 / 14 | 29% |
| ② 계층 추론 | **1 / 14** (핵심), 3 (부수) | **7%** |

#### Step 3. **데이터 모델 선택** — RDF인가 LPG인가

Step 2의 분포가 결론을 강제한다.

---

##### 판단 근거 1 — 계층 추론이 거의 없다

**RDF/OWL이 LPG 대비 갖는 유일한 구조적 강점은 표준 추론(subsumption)이다.** 그런데 CQ 14개 중 **계층 추론을 핵심으로 요구하는 것은 CQ8 하나뿐**이다(계약자 유형 계층). 그마저도 계층 깊이가 2~3단계다:

```
계약자
 ├─ 개인
 │   ├─ 성년
 │   └─ 미성년
 ├─ 법인
 └─ (교차) 외국인
```

이 정도는 Cypher 가변길이 패턴으로 충분히 대체된다:

```cypher
MATCH (c:계약자)-[:IS_A*0..3]->(:계약자유형 {name:"미성년"})
RETURN c
```

**추론기를 도입할 만한 이득이 없다.**

---

##### 판단 근거 2 — 압도적 다수가 조건부 규칙인데, OWL은 규칙 엔진이 아니다

**이게 핵심 논점이다.**

OWL은 **Description Logic** 기반이다. "이 개체가 어떤 클래스에 속하는가"(분류)에 특화되어 있다. **if-then 업무 규칙 엔진이 아니다.**

CQ5를 보자:
```
IF 계약상태 = 실효
   AND 실효경과일 ≤ 30일
   AND 미납보험료 완납
   AND 부활청약서 제출
THEN 자동부활 대상
```

이걸 OWL로 표현하려면 SWRL 규칙을 얹어야 하고, 수치 비교(`≤ 30일`)는 built-in을 써야 하며, 추론기 성능이 급락한다. **복잡도 대비 이득이 없다.**

**즉, RDF를 골라도 어차피 OWL 밖(SWRL/SHACL rules/애플리케이션 코드)에서 규칙을 처리해야 한다. 그렇다면 RDF의 강점이 발휘될 자리가 없다.**

---

##### 판단 근거 3 — 관계 속성 요구가 절반이다

CQ2, CQ4, CQ6, CQ9, CQ11, CQ12가 **엣지 자체에 조건을 달아야** 자연스럽다.

CQ6 "계약상태 × 변경유형 → 허용/불허"를 보자.

**LPG:**
```cypher
(계약변경_감액)-[:허용됨 {계약상태:"정상", 조건:"납입 2회 이상"}]->(신청가능)
(계약변경_감액)-[:불허됨 {계약상태:"실효", 사유:"부활 선행 필요"}]->(신청불가)
```
**한 줄. 엣지에 조건이 그대로 붙는다.**

**RDF:**
```turtle
aia:규칙_감액_001 a aia:변경허용규칙 ;
    aia:대상변경   aia:계약변경_감액 ;
    aia:계약상태   aia:정상 ;
    aia:조건       "납입 2회 이상" ;
    aia:결과       aia:신청가능 .
```
**관계를 노드로 승격(reification)해야 한다.** 이런 규칙이 수백 개면 트리플이 폭증하고, 쿼리도 그만큼 길어진다.

___

##### 판단 근거 4 — 다중홉 순회 성능

CQ13(파급 효과 추적)은 계약 → 특약 → 부가서비스 → 정산 순으로 여러 단계를 탄다. **Neo4j의 Index-Free Adjacency가 정확히 이 패턴에 최적화되어 있다.** 트리플스토어는 SPARQL 조인으로 처리하며, 깊은 순회에서 상대적으로 느리다.

---

##### 판단 근거 5 — 전역 데이터 통합 요구가 없다

RDF의 URI 기반 통합은 **여러 조직/시스템의 데이터를 합칠 때** 빛난다. 우리는 **AIA 사내 문서 21개**를 다룬다. 외부 온톨로지와 병합할 계획이 없다. **URI의 이점이 발생하지 않는다.**

---

##### **결론**

> **LPG(Neo4j)를 선택한다.**
>
> **근거:**
> - 계층 추론 요구 7% (1/14) → RDF의 유일한 강점이 발휘될 자리가 없음
> - 조건부 규칙 요구 93% (13/14) → OWL은 규칙 엔진이 아니므로, RDF를 써도 규칙은 밖에서 처리해야 함
> - 관계 속성 요구 50% (7/14) → LPG 네이티브, RDF는 reification으로 트리플 폭증
> - 얕은 계층(2~3단계)은 `*0..` 가변길이 패턴으로 충분히 대체 가능
> - 조직 간 데이터 통합 요구 없음 → URI의 이점 미발생
>
> **버리는 것 :**
> - 표준 추론기(HermiT/Pellet) 사용 불가 → 계층 확장은 쿼리에 명시해야 함
> - SHACL 수준의 형식 검증 불가 → Neo4j 제약조건(unique/exists/node key) + 애플리케이션 검증으로 대체
> - W3C 표준 이식성 → 향후 RDF 전환이 필요해지면 neosemantics(n10s)로 마이그레이션 경로 존재
>
> **재검토 조건:** 향후 다른 도메인 파일(보상/신계약 등)이 합쳐지면서 **깊은 상품 분류 계층**이나 **외부 온톨로지(FIBO) 통합** 요구가 생기면 RDF를 재검토한다.

이 결정은 "CQ 분포"에 근거해서 결정한 내용이다.

---

#### Step 4. 개념 모델링 — **형식 중립**

여기부터가 실제 온톨로지 설계다. **아직 Turtle도 Cypher도 쓰지 않는다.** 종이 위 작업이다.

##### 4-1. CQ에서 명사/동사 추출

```
CQ3: "이 계약변경 건에 타인을위한해지프로세스가 적용되는가?
      (계약자와 수익자가 상이한 경우)"

명사 → Class 후보:  계약변경, 보험계약, 계약자, 수익자, 프로세스
동사 → Property 후보: (보험계약)-계약자→(사람)
                     (보험계약)-수익자→(사람)
                     (계약변경)-적용프로세스→(프로세스)
조건 → Axiom/Rule:   계약자 ≠ 수익자 → 타인을위한해지프로세스 적용
```

**14개 CQ 전부에 이 작업을 반복한다.** 여기서 나온 명사/동사가 온톨로지의 원재료다.

---

##### 4-2. 역할(Role) vs 본질(Essence) 판별 — **가장 중요한 설계 판단**

**문제:** 홍길동 한 사람이 **계약자이면서 동시에 피보험자**일 수 있다. `계약자`와 `피보험자`를 클래스로 만들면 어떻게 되는가?

**좋지 않은 설계:**
```
Class: 계약자, 피보험자, 수익자
→ 홍길동은 계약자 클래스이면서 피보험자 클래스에 동시 소속?
```

**왜 안좋은가:** "계약자"는 사람의 **본질이 아니라 역할**이다. 홍길동이 계약을 해지하면 계약자가 아니게 되지만, **사람이기를 그만두지는 않는다.** 역할은 계약과의 **관계에서 파생**된다. 사람 자체의 속성이 아니다.

**판단 기준:**

> **그것이 사라져도 개체가 계속 존재하면 → 관계(Property/Relationship)**
> **그것이 사라지면 개체도 사라지면 → 클래스(Class/Label)**

| 후보 | 사라지면? | 판정 |
|---|---|---|
| 계약자 | 홍길동은 여전히 존재 | **관계** |
| 피보험자 | 홍길동은 여전히 존재 | **관계** |
| 수익자 | 홍길동은 여전히 존재 | **관계** |
| 사람 | 홍길동이 존재하지 않음 | **클래스** |
| 종신보험 | 그 계약은 종신보험이 아님 (계약 정체성이 바뀜) | **클래스** |
| 실효(계약상태) | 계약은 여전히 존재, 상태만 바뀜 | **관계 또는 속성** ← 아래 참조 |

**올바른 설계 (개념 층):**
```
Class: 사람, 보험계약, 계약변경, 서류, 채널, 계약상태, 프로세스

Relationship:
  (보험계약) --계약자--> (사람)
  (보험계약) --피보험자--> (사람)     ← 같은 사람 가능
  (보험계약) --수익자--> (사람)
  (보험계약) --현재상태--> (계약상태)
```

---

##### 4-3. 상태(State)를 어떻게 다룰 것인가 — CQ5, CQ14가 강제하는 결정

**CQ14가 결정적이다:**
> "청약 시점과 현재 시점의 상태가 다를 경우 어느 시점 기준으로 판단하는가?"

**이 질문은 "계약상태를 단순 속성으로 두면 답할 수 없다"는 뜻이다.**

```
좋지 않은 설계: (계약 {상태: "실효"})
   → 언제부터 실효인지, 이전 상태가 뭐였는지 모름. CQ5(실효경과일), CQ14(시점기준) 답 불가

좋은 설계: 상태를 관계로 승격 + 엣지에 시점 부착
   (계약)-[:상태이력 {시작일:"2024-03-01", 종료일:null}]->(:계약상태 {name:"실효"})
   (계약)-[:상태이력 {시작일:"2023-01-01", 종료일:"2024-02-29"}]->(:계약상태 {name:"정상"})
```

**여기가 LPG를 고른 이득이 직접 발생하는 지점이다.** RDF였다면 `상태이력` 관계에 시작일/종료일을 못 붙이므로 reification이 필요하다.

---

##### 4-4. 규칙(Rule)을 그래프에 넣을 것인가, 밖에 둘 것인가

**CQ 13/14가 조건부 규칙이므로, 이 결정이 아키텍처를 가른다.**

| 방식 | 설명 | 장점 | 단점 |
|---|---|---|---|
| **A. 규칙을 노드로** | 규칙 자체를 그래프에 넣고, 조건을 엣지 속성으로 | 규칙 조회/설명 가능, LLM이 근거 제시 가능 | 복잡한 조건 표현 한계 |
| **B. 규칙은 코드로** | 그래프는 사실만, 규칙은 애플리케이션에서 | 표현력 무제한 | 규칙이 그래프 밖 → RAG가 근거를 못 댐 |

**RAG 목적상 A를 우선한다.** 이유: LLM이 "왜 그런 판정이 나왔는지"를 설명하려면, **규칙 자체가 검색 가능해야** 한다. 코드에 숨어 있으면 근거를 못 댄다.

```
(:변경규칙 {id:"R-감액-001", 설명:"실효 상태에서는 감액 불가, 부활 선행 필요"})
   -[:적용대상]-> (:계약변경 {name:"감액"})
   -[:전제조건]-> (:계약상태 {name:"실효"})
   -[:판정]-> (:결과 {value:"불허"})
   -[:근거문서]-> (:문서 {file:"계약변경_03.pdf", 조항:"제4조 2항"})
```

**`근거문서` 관계가 핵심이다.** 이게 있어야 그래프 답변에 **원문 출처**를 붙일 수 있고, 벡터 검색과 연결된다.

#### Step 5. 구현 — Cypher (LPG)

Step 3에서 LPG를 골랐으므로, Step 4의 개념 모델을 Cypher 스키마로 내린다.

```cypher
// ── 제약조건 (RDF의 SHACL 역할을 여기서 대체) ──
CREATE CONSTRAINT 계약_고유 IF NOT EXISTS
  FOR (c:보험계약) REQUIRE c.계약번호 IS UNIQUE;

CREATE CONSTRAINT 사람_고유 IF NOT EXISTS
  FOR (p:사람) REQUIRE p.고객번호 IS UNIQUE;

CREATE CONSTRAINT 규칙_필수 IF NOT EXISTS
  FOR (r:변경규칙) REQUIRE r.근거문서 IS NOT NULL;
```

```cypher
// ── 인스턴스 예시 ──
CREATE (c:보험계약:생명보험:종신보험 {계약번호:"2024-001", 청약일:date("2024-01-15")})
CREATE (h:사람 {고객번호:"C001", 이름:"홍길동", 생년:1985})
CREATE (k:사람 {고객번호:"C002", 이름:"김영희"})

CREATE (c)-[:계약자]->(h)
CREATE (c)-[:피보험자]->(h)      // 같은 사람 — 역할이 관계이므로 자연스러움
CREATE (c)-[:수익자]->(k)         // 다름 → CQ3 발동 (타인을위한해지)

CREATE (정상:계약상태 {name:"정상"})
CREATE (실효:계약상태 {name:"실효"})
CREATE (c)-[:상태이력 {시작일:date("2024-01-15"), 종료일:date("2025-02-28")}]->(정상)
CREATE (c)-[:상태이력 {시작일:date("2025-03-01"), 종료일:null}]->(실효)
```

##### CQ가 실제로 답이 되는지 검증

**CQ3 — 타인을위한해지 적용 여부:**
```cypher
MATCH (c:보험계약 {계약번호:"2024-001"})
MATCH (c)-[:계약자]->(pc:사람)
MATCH (c)-[:수익자]->(bf:사람)
RETURN c.계약번호,
       pc.고객번호 <> bf.고객번호 AS 타인을위한해지_적용
```

**CQ5 — 자동부활 대상 여부 (조건부 규칙):**
```cypher
MATCH (c:보험계약 {계약번호:"2024-001"})
      -[s:상태이력]->(:계약상태 {name:"실효"})
WHERE s.종료일 IS NULL
WITH c, duration.inDays(s.시작일, date()).days AS 실효경과일
MATCH (r:변경규칙)-[:적용대상]->(:계약변경 {name:"부활"})
MATCH (r)-[:근거문서]->(doc:문서)
RETURN c.계약번호, 실효경과일,
       실효경과일 <= 30 AS 자동부활_대상,
       r.설명, doc.file, doc.조항       // ← 근거까지 함께 반환
```

**`doc.file`, `doc.조항`이 함께 나오는 게 핵심이다.** 이게 LLM 컨텍스트로 들어가서 "제4조 2항에 따라..."라는 답변이 가능해진다.

**CQ8 — 미성년 계약자 추가절차 (계층 추론 대체):**
```cypher
// OWL 추론기 없이 가변길이 패턴으로 계층 확장
MATCH (c:보험계약 {계약번호:"2024-001"})-[:계약자]->(p:사람)
MATCH (p)-[:유형]->(t:계약자유형)-[:IS_A*0..3]->(:계약자유형 {name:"미성년"})
MATCH (rule:추가절차규칙)-[:대상유형]->(:계약자유형 {name:"미성년"})
MATCH (rule)-[:필요서류]->(doc:서류)
RETURN p.이름, collect(doc.name) AS 추가필요서류
```

**`*0..3`이 OWL의 `subClassOf` 추론을 대체하는 지점이다.** 이걸 명시적으로 설명할 수 있어야 한다.

#### Step 7. 문서 → 인스턴스 추출 (LLM)

**스키마(TBox)가 확정된 후**, LLM에게 스키마를 주고 21개 파일에서 인스턴스(ABox)를 뽑는다.

```
[시스템 프롬프트에 스키마 명시]

다음 Label과 Relationship만 사용하여 문서에서 그래프를 추출하라.
목록에 없는 것은 절대 만들지 마라.

Labels:  보험계약, 사람, 계약변경, 계약상태, 서류, 채널, 
         변경규칙, 계약자유형, 문서, 조항
Relationships: 계약자, 피보험자, 수익자, 상태이력, 적용대상, 
               전제조건, 판정, 근거문서, 필요서류, IS_A

출력 형식: Cypher CREATE 문
필수: 모든 :변경규칙 노드는 반드시 -[:근거문서]-> 를 가져야 한다.
      근거문서에는 파일명과 조항 번호를 반드시 명시하라.
```

**스키마를 주는 것과 안 주는 것의 차이가 결정적이다.**
- 스키마 없이 추출 → "계약자" / "보험계약자" / "가입자"가 **다 다른 노드**가 됨. 그래프가 파편화되어 쓸모없어짐
- 스키마 강제 → 21개 파일에서 나온 엔티티가 **하나로 병합**됨

**추출 후 검증 (SHACL 대체):**
```cypher
// 근거문서 없는 규칙 = 추출 실패
MATCH (r:변경규칙) WHERE NOT (r)-[:근거문서]->() 
RETURN r.id AS 근거누락_규칙;

// 고아 노드 = 관계 추출 실패
MATCH (n) WHERE NOT (n)--() RETURN labels(n), count(*);

// 스키마 밖 라벨 = 프롬프트 위반
MATCH (n) WHERE NOT any(l IN labels(n) 
  WHERE l IN ['보험계약','사람','계약변경','계약상태','서류','채널',
              '변경규칙','계약자유형','문서','조항'])
RETURN DISTINCT labels(n);
```

#### Step 8. 최종 아키텍처 — 하이브리드

**그래프가 벡터를 대체하는 게 아니다. 역할이 다르다.**

```
                        질문
                          │
          ┌───────────────┴───────────────┐
          │                               │
   [Vector Search]                  [Graph Query]
    관련 문서 청크                  구조적 사실 + 근거출처
   "약관 원문 그대로"           "계약자≠수익자 → 타인을위한해지"
   "제4조 2항 전문"              "실효경과 28일 → 자동부활 대상"
          │                               │
          └───────────────┬───────────────┘
                          │
                    LLM 컨텍스트
                          │
                       최종 답변
              (사실 판정 + 원문 근거 인용)
```

| | Vector | Graph |
|---|---|---|
| **역할** | **어디에 그렇게 쓰여 있는가** (근거) | **무엇이 참인가** (사실) |
| **강점** | 원문 인용, 표현 다양성 흡수 | 관계 정확성, 완전성, 다중홉 |
| **약점** | 관계 구분 못함, 완전성 없음 | 원문 뉘앙스 손실 |

**연결 고리는 `근거문서` 관계다.** 그래프가 규칙을 판정하면서 `(파일명, 조항번호)`를 함께 반환하고, 그걸로 벡터 스토어에서 해당 청크를 정확히 끌어온다. **그래프가 벡터의 라우터 역할을 한다.**

###  논증 구조

#### ① 층위 분리 (혼동 제거)

> "온톨로지·RDF·Neo4j는 같은 층위가 아닙니다.
> **온톨로지는 개념 모델** — 무엇이 존재하고 어떻게 연결되는가.
> **RDF와 LPG는 데이터 모델** — 그걸 어떤 자료구조로 담는가.
> **Neo4j·Neptune·GraphDB는 저장 엔진** — 그 자료구조를 어떻게 디스크에 굽는가.
>
> 이 셋은 **분리 가능한 결정**이고, 저는 실제로 분리해서 진행했습니다."

#### ② RDF의 본질과 대가

> "RDF는 **(주어, 서술어, 목적어) 트리플 하나가 전부**입니다. 그래프는 트리플을 던져 넣으면 저절로 생기는 부산물입니다.
>
> 대가가 있습니다. 트리플은 3칸뿐이라 **엣지에 속성을 못 답니다.** '2024년 1월 1일에 체결했다'를 쓰려면 관계를 노드로 승격시켜야 하고(reification), 트리플 1개가 4개로 늘어납니다."

#### ③ RDF의 유일한 강점, 그리고 그 조건 (**실력 판별점**)

> "RDF가 LPG 대비 갖는 **유일한 구조적 강점은 표준 추론**입니다. `종신보험 subClassOf 생명보험`을 선언하면 추론기가 '계약_001은 생명보험이다'를 **자동 생성**합니다. 계층을 쿼리에 하드코딩할 필요가 없어집니다.
>
> 단, 이건 **Open World Assumption** 위에서 돌아갑니다. 그래서 `rdfs:range`는 **제약이 아니라 추론 규칙**입니다. 계약자의 range를 '사람'으로 선언하고 법인을 넣으면, 에러가 아니라 **'법인도 사람이다'라고 추론합니다.**
>
> 보험 데이터에서 이건 위험합니다. **RDF를 쓰려면 SHACL로 검증을 따로 걸어야 합니다.** RDFS/OWL만으로는 데이터 품질을 잡을 수 없습니다."

#### ④ Neo4j 물리 구조 (**이사님이 명시적으로 물은 것**)

> "Neo4j가 빠른 이유는 **Index-Free Adjacency**입니다.
>
> RDBMS는 JOIN마다 B-tree를 타서 홉당 **O(log N)**입니다. Neo4j는 **노드 레코드 15바이트, 관계 레코드 34바이트로 크기를 고정**해서, ID만 알면 `ID × 15`로 디스크 오프셋을 **곱셈 한 번에 계산**합니다. 관계 레코드가 이중 연결 리스트를 이루므로, 이웃으로 가는 건 **인덱스 없이 포인터를 따라가는 O(1)** 연산입니다. **로컬 순회 비용이 전체 데이터 크기와 무관합니다.**
>
> 단서 둘. **첫 노드는 인덱스를 탑니다** — IFA는 그 다음부터입니다. 그리고 관계가 수백만 개 붙은 **슈퍼노드**에서는 연결 리스트가 길어져 이점이 무너집니다. 이건 엔진이 아니라 **모델링으로 피해야 합니다.**"

#### ⑤ 왜 LPG인가 (**여기가 진짜 평가받는 곳 — 표를 보여준다**)

> "**CQ 14개를 도출하고 유형별로 분류했습니다.**
>
> - **조건부 규칙: 13/14 (93%)**
> - 관계 속성 요구: 7/14 (50%)
> - 다중홉 순회: 6/14 (43%)
> - **계층 추론: 1/14 (7%)**
>
> **RDF의 유일한 강점인 계층 추론 요구가 7%입니다.** 그마저도 깊이 2~3단계라 Cypher 가변길이 패턴 `*0..3`으로 대체됩니다.
>
> 반면 **93%가 조건부 규칙**인데, **OWL은 규칙 엔진이 아닙니다.** Description Logic 기반이라 분류(subsumption)에 특화되어 있지, if-then 규칙은 SWRL이나 SHACL rules를 따로 얹어야 하고 성능이 급락합니다. **RDF를 골라도 어차피 규칙은 OWL 밖에서 처리해야 합니다. 그럼 RDF의 강점이 발휘될 자리가 없습니다.**
>
> 게다가 절반의 CQ가 **엣지에 조건을 달아야** 자연스럽습니다. LPG는 네이티브고, RDF는 reification으로 트리플이 폭증합니다.
>
> **그래서 LPG를 선택했습니다. 유행이 아니라 CQ 분포에 근거한 선택입니다.**
>
> **대신 버리는 것도 명확합니다.** 표준 추론기와 SHACL 형식 검증을 못 씁니다. 계층 확장은 쿼리에 명시해야 하고, 검증은 Neo4j 제약조건과 애플리케이션 레벨로 대체합니다. **향후 다른 도메인 파일이 합쳐지며 깊은 상품 계층이 생기거나 FIBO 통합 요구가 나오면, neosemantics로 RDF 전환 경로를 열어둘 수 있습니다.**"

#### ⑥ 모델링 원칙 (설계 감각)

> "설계에서 가장 중요한 판단은 **무엇이 클래스이고 무엇이 관계인가**였습니다.
>
> '계약자'를 클래스로 만들면 안 됩니다. 계약자는 사람의 **본질이 아니라 역할**입니다. 홍길동이 계약을 해지해도 홍길동은 존재합니다. **역할은 관계로 모델링**했고, 그래서 홍길동이 계약자이면서 동시에 피보험자인 경우가 자연스럽게 표현됩니다.
>
> 기준은 하나입니다. **그것이 사라져도 개체가 존속하면 관계, 개체도 사라지면 클래스.**
>
> 같은 논리로, **계약상태도 단순 속성이 아니라 관계로 승격**시켰습니다. CQ14가 '청약 시점과 현재 시점 중 어느 쪽 기준인가'를 묻기 때문입니다. `(계약)-[:상태이력 {시작일, 종료일}]->(:실효)` 형태로 **엣지에 시점을 달았고, 이건 LPG를 고른 이득이 직접 발생하는 지점입니다.** RDF였다면 여기서 reification이 필요했습니다."

#### ⑦ 벡터를 버리지 않는 이유

> "그래프로 벡터를 대체하는 게 아닙니다. **역할이 다릅니다.**
>
> - **그래프**: **무엇이 참인가.** '계약자 ≠ 수익자 → 타인을위한해지 적용'은 유사도로 안 나옵니다.
> - **벡터**: **어디에 그렇게 쓰여 있는가.** 약관 원문을 그대로 인용해야 하는 상황에 필요합니다.
>

#### 지금 당장 해야 할 것 (우선순위)
